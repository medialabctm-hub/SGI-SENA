DOCUMENTACIÓN BASE DE DATOS - GESTIÓN DE EQUIPOS SENA
================================================================================

VERSIÓN: 1.0 Normalizada

================================================================================
1. DESCRIPCIÓN GENERAL
================================================================================

La base de datos GestionEquipo está diseñada para gestionar el inventario de 
equipos tecnológicos del SENA, incluyendo su asignación, mantenimiento, 
verificación y responsabilidades. El sistema soporta múltiples roles de usuario
(Administrador, Instructor, Aprendiz) y gestiona ambientes físicos, clases 
programadas y verificaciones de inventario.

2. ESTRUCTURA DE TABLAS
================================================================================

2.1. TABLAS DE USUARIOS Y SEGURIDAD
------------------------------------

Roles
-----
Almacena los roles del sistema con sus descripciones.
- id_rol: Identificador único del rol
- nombre_rol: Nombre del rol (Administrador, Instructor, Aprendiz)
- descripcion: Descripción del rol
- fecha_creacion: Fecha de creación del registro

Usuarios
--------
Gestiona los usuarios del sistema con sus credenciales y datos personales.
- id_usuario: Identificador único del usuario
- nombre_usuario: Nombre completo del usuario
- cedula: Cédula única del usuario
- telefono: Teléfono de contacto
- correo: Correo electrónico único
- area_usuarios: Área o departamento del usuario
- contrasena: Contraseña encriptada
- id_rol: Referencia al rol del usuario
- estado: Estado del usuario (Activo, Inactivo)
- requiere_cambio_contrasena: Indica si debe cambiar la contraseña
- fecha_registro: Fecha de registro en el sistema
- ultimo_acceso: Última fecha de acceso
- creado_por: Usuario que creó este registro

Tokens_Recuperacion_Contrasena
-------------------------------
Gestiona tokens temporales para recuperación de contraseñas.
- id_token: Identificador único del token
- id_usuario: Usuario asociado al token
- token: Token único de recuperación
- fecha_creacion: Fecha de creación del token
- fecha_expiracion: Fecha de expiración del token
- usado: Indica si el token ya fue utilizado
- fecha_uso: Fecha en que se usó el token

Notificaciones
--------------
Sistema de notificaciones para usuarios.
- id_notificacion: Identificador único de la notificación
- id_usuario: Usuario destinatario
- titulo: Título de la notificación
- cuerpo: Contenido de la notificación
- tipo: Tipo de notificación (info, aviso, alerta, critica)
- leida: Indica si fue leída
- fecha_creacion: Fecha de creación
- fecha_lectura: Fecha en que fue leída
- metadata: Datos adicionales en formato JSON
- creado_por: Usuario que creó la notificación

Preferencias_Usuario
--------------------
Almacena preferencias de configuración de cada usuario.
- id_preferencia: Identificador único
- id_usuario: Usuario asociado (único)
- notificaciones_email: Preferencia de notificaciones por email
- notificaciones_sms: Preferencia de notificaciones por SMS
- notificaciones_app: Preferencia de notificaciones en la app
- idioma: Idioma de la interfaz (es, en, etc.)
- zona_horaria: Zona horaria del usuario
- fecha_creacion: Fecha de creación
- fecha_actualizacion: Fecha de última actualización

Invitation_Codes
----------------
Códigos de invitación para registro de usuarios.
- id_codigo: Identificador único del código
- codigo: Código de invitación único
- rol_destinado: Rol para el cual es válido el código
- fecha_creacion: Fecha de creación
- fecha_expiracion: Fecha de expiración
- max_usos: Número máximo de usos (0 = ilimitado)
- usos_actuales: Número de veces usado
- estado: Estado del código (Activo, Inactivo, Expirado, Agotado)
- creado_por: Usuario que creó el código

4. TRIGGERS
================================================================================

trg_historial_estado_inicial
-----------------------------
Registra en historial cuando se crea un estado inicial de equipo.

trg_historial_cambio_estado
----------------------------
Registra en historial cuando cambia el estado operativo de un equipo.

trg_historial_cambio_ambiente
------------------------------
Registra en historial cuando un equipo cambia de ambiente.

trg_estado_mantenimiento
------------------------
Actualiza el estado del equipo a "En Mantenimiento" cuando se crea un 
mantenimiento con estado "En Proceso".

trg_finalizar_responsabilidades
--------------------------------
Finaliza responsabilidades activas cuando se elimina un equipo.

trg_validar_imagen_principal_equipo
------------------------------------
Asegura que solo haya una imagen principal por equipo.

trg_validar_imagen_principal_ambiente
--------------------------------------
Asegura que solo haya una imagen principal por ambiente.

trg_finalizar_responsabilidades_ambiente_anterior
--------------------------------------------------
Finaliza responsabilidades anteriores cuando inicia una nueva clase o cuando 
una clase finaliza.


================================================================================
5. PROCEDIMIENTOS ALMACENADOS
================================================================================

sp_asignar_responsables
------------------------
Asigna múltiples responsables a un equipo y actualiza su estado a "En Uso".
Parámetros:
- p_codigo_equipo: Código del equipo
- p_usuarios_json: JSON con array de usuarios a asignar
- p_asignado_por: Usuario que realiza la asignación

sp_devolver_equipo
------------------
Devuelve un equipo, finaliza responsabilidades y actualiza estado a "Disponible".
Parámetros:
- p_codigo_equipo: Código del equipo
- p_actualizado_por: Usuario que realiza la devolución

sp_asignar_automatico
---------------------
Asigna automáticamente un equipo disponible según criterios.
Parámetros:
- p_id_usuario: Usuario al que se asigna
- p_id_ambiente: Ambiente preferido (puede ser NULL)
- p_tipo_equipo: Tipo de equipo preferido (puede ser NULL)
- p_asignado_por: Usuario que realiza la asignación

sp_historial_equipo
-------------------
Obtiene el historial completo de un equipo.
Parámetros:
- p_codigo_equipo: Código del equipo

sp_equipos_por_usuario
----------------------
Obtiene todos los equipos asignados a un usuario.
Parámetros:
- p_id_usuario: ID del usuario

sp_mover_equipo_ambiente
------------------------
Mueve un equipo a otro ambiente y registra el movimiento.
Parámetros:
- p_codigo_equipo: Código del equipo
- p_id_ambiente_nuevo: Nuevo ambiente
- p_observaciones: Observaciones del movimiento
- p_actualizado_por: Usuario que realiza el movimiento

sp_reportar_novedad
-------------------
Reporta una novedad sobre un equipo y opcionalmente actualiza su estado.
Parámetros:
- p_codigo_equipo: Código del equipo
- p_tipo_novedad: Tipo de novedad
- p_descripcion: Descripción de la novedad
- p_reportado_por: Usuario que reporta
- p_actualizar_estado: Si actualizar el estado a "Dañado"

sp_estadisticas_sistema
-----------------------
Obtiene estadísticas generales del sistema.
No requiere parámetros.

sp_iniciar_clase
----------------
Inicia una clase, asigna responsabilidades y finaliza responsabilidades anteriores.
Parámetros:
- p_id_clase: ID de la clase
- p_fecha_inicio_real: Fecha y hora real de inicio

sp_finalizar_clase
------------------
Finaliza una clase y sus responsabilidades asociadas.
Parámetros:
- p_id_clase: ID de la clase
- p_fecha_fin_real: Fecha y hora real de fin (puede ser NULL)

sp_responsables_ambiente_actual
--------------------------------
Obtiene los responsables actuales de un ambiente en una fecha específica.
Parámetros:
- p_id_ambiente: ID del ambiente
- p_fecha_consulta: Fecha de consulta (puede ser NULL para usar NOW())

- Elementos -> Categorias_Equipo (muchos a uno)
- Elementos -> Ambientes (muchos a uno)
- Elementos -> Estado_Equipo (uno a uno)
- Responsables_Equipo -> Elementos (muchos a uno)
- Responsables_Equipo -> Usuarios (muchos a uno)
- Clases -> Ambientes (muchos a uno)
- Clases -> Usuarios (instructor, muchos a uno)
- Participantes_Clase -> Clases (muchos a uno)
- Participantes_Clase -> Usuarios (muchos a uno)
- Responsabilidades_Ambiente -> Ambientes (muchos a uno)
- Responsabilidades_Ambiente -> Clases (muchos a uno, opcional)
- Responsabilidades_Ambiente -> Usuarios (muchos a uno)
- Mantenimiento -> Elementos (muchos a uno)
- Novedades -> Elementos (muchos a uno)
- Historial_Equipos -> Elementos (muchos a uno)
- Componentes_Asociados -> Elementos (muchos a uno, dos relaciones)

================================================================================
9. NOTAS DE IMPLEMENTACIÓN
================================================================================

- Todos los campos de fecha usan DATETIME para mayor precisión
- Los estados se manejan mediante ENUM para garantizar consistencia
- Se usa ON DELETE CASCADE en relaciones donde tiene sentido eliminar registros relacionados
- Se usa ON DELETE RESTRICT en relaciones críticas (ej: Elementos -> Ambientes)
- Los triggers automáticamente mantienen el historial actualizado
- Las vistas proporcionan consultas optimizadas para reportes comunes
- Los procedimientos almacenados encapsulan lógica de negocio compleja


================================================================================
10. MEJORAS IMPLEMENTADAS EN NORMALIZACIÓN
================================================================================

- Consolidación de definiciones de tablas (eliminación de ALTER TABLE redundantes)
- Optimización de índices (eliminación de duplicados, consolidación de compuestos)
- Simplificación de triggers (código más limpio y eficiente)
- Optimización de procedimientos almacenados (eliminación de código duplicado)
- Reorganización de comentarios y documentación inline
- Reducción de código excedente sin afectar funcionalidad
